Quiero que construyas la base arquitectónica completa del nuevo sitio web corporativo de iData dentro de Figma Make. Este proyecto debe incluir dos módulos dentro del mismo entorno: el sitio público y un administrador CMS interno totalmente funcional a nivel estructural.

Este no es un ejercicio visual. Es una fundación técnica. Prioriza arquitectura, modelo de datos, escalabilidad, mantenibilidad del código y preparación para conexión a base de datos real.

El sitio debe funcionar estructuralmente en dos idiomas reales: español e inglés. No es un simple toggle. Deben existir rutas independientes para cada idioma. Todas las rutas deben comenzar con /es/ para español y /en/ para inglés. Cada idioma debe tener sus propios slugs, su propio contenido y su propia metadata SEO.

El Home en español y el Home en inglés deben poder ser completamente diferentes en estructura, orden y secciones visibles. El administrador debe permitir configurar qué secciones aparecen en cada idioma y en qué orden.

La arquitectura oficial del sitio que debes implementar es la siguiente:

Home:
- /es/
- /en/

Servicios:
- /es/servicios/
- /en/services/

Categorías de servicios tipo cluster SEO:
- strategy-consulting
- data-delivery
- data-operations
- cloud-services

Cada subservicio debe tener landing individual optimizada para SEO:
- /es/servicios/{slug-es}
- /en/services/{slug-en}

Industrias:
- /es/industrias/
- /en/industries/
- /es/industrias/{slug-es}
- /en/industries/{slug-en}

Casos de éxito:
- /es/casos/
- /en/case-studies/
- /es/casos/{slug-es}
- /en/case-studies/{slug-en}

Insights (Blog):
- /es/insights/
- /en/insights/
- /es/insights/{slug-es}
- /en/insights/{slug-en}

Nosotros:

Talento:

- /es/talento/ofertas/

Ahora construye dos grandes módulos dentro del proyecto:

1) PUBLIC SITE

Debes crear:
- Layout global reutilizable con Header, Footer y contenedor responsive
- Sistema de navegación multi-idioma
- Componente SEO global reutilizable
- Breadcrumb component
- Estructura de páginas lista para consumir datos desde colecciones
- Mock data inicial para probar renderizado

Las páginas deben existir a nivel estructural, aunque sean wireframe minimal sin diseño final.

2) ADMIN CMS

Debes construir un administrador con layout propio (sidebar + topbar) que permita crear, editar y gestionar las siguientes colecciones:

Services
ServiceCategories
Industries
CaseStudies
BlogPosts
BlogCategories
Jobs
TeamMembers
Resources
Testimonials
HomeSections

Cada colección debe tener campos separados para español e inglés incluyendo:

- title_es y title_en
- slug_es y slug_en
- contenido rico para cada idioma
- resumen para cada idioma cuando aplique
- metadata SEO independiente por idioma (meta title, meta description, og image)
- imagen destacada
- estado (draft o published)
- campo featured cuando aplique
- orden manual
- fecha cuando aplique

La colección HomeSections es crítica. Debe permitir:
- Definir idioma (ES o EN)
- Tipo de sección (hero, servicios destacados, industrias destacadas, casos destacados, insights destacados, testimonios, logos, CTA, estadísticas, sección personalizada)
- Activar o desactivar sección
- Definir orden
- Referenciar otras colecciones (por ejemplo seleccionar 3 casos destacados)
- Configurar título, subtítulo y CTA por idioma

El código debe estar estructurado por capas separadas para sitio público y administrador. Usa componentes reutilizables. Organiza todo pensando en que un desarrollador pueda conectar base de datos real fácilmente.

Optimiza todo desde el inicio para SEO técnico:
- Soporte para metadata dinámica por página
- Canonical
- Hreflang entre ES y EN
- Estructura correcta de headings
- Base preparada para implementar schema estructurado según tipo de página

En esta fase no quiero branding final. Quiero arquitectura, rutas funcionando, colecciones definidas, layouts globales y CMS estructuralmente sólido.

Prioriza mantenibilidad, escalabilidad y calidad de arquitectura sobre estética visual.
